# 嵌入式产品研发C语言编码规范要求

为规范嵌入式产品研发C语言编程行为，提升代码可读性、可维护性、可移植性及安全性，降低团队协作成本，规避研发过程中的常见编码风险，结合部门嵌入式产品（基于C语言）的研发特性，特制定本编码规范，所有参与嵌入式产品研发的同事必须严格遵照执行。本规范重点明确axk函数头要求，同时覆盖编码全流程核心要点，后续将根据产品迭代及研发需求逐步补充完善。

本规范适用于部门所有嵌入式产品的C语言代码编写、修改、评审及维护工作，包括但不限于底层驱动、应用层逻辑、工具类代码等，无论是新开发代码还是历史代码优化，均需符合本规范要求。

## 一、核心要求：axk函数头规范（必严格执行）

所有C语言函数（包括全局函数、静态函数、中断服务函数、工具函数等，无例外），必须在函数定义及声明的开头，添加以“axk”为前缀的函数头注释，函数头需清晰、完整，涵盖核心信息，便于他人理解函数功能、用途及使用方式，具体规范如下：

### 1\.1 函数头格式（固定模板）

```c
/*
 * axk_函数功能简述（简洁明了，1句话概括，不超过50字）
 * @brief  函数详细描述（补充功能细节、设计思路，可选，复杂函数必写）
 * @param  参数名1：参数说明（明确参数类型、含义、取值范围，无参数则写“无”）
 * @param  参数名2：参数说明（多个参数依次排列，参数名与函数定义一致）
 * @return 返回值说明（明确返回值类型、含义、不同返回值对应的场景，无返回值则写“无”）
 * @note   注意事项（可选，如函数使用限制、调用前提、异常处理说明、与其他函数的关联等）
 * @author 开发者姓名
 * @date   开发/修改日期（格式：YYYY-MM-DD）
 */
返回值类型 axk_函数名(参数列表)
{
    // 函数体
}
```

### 1\.2 具体要求

- 函数头前缀“axk\_”必须与函数名前缀保持一致，即函数名必须以“axk\_”开头，函数头注释首行必须包含“axk\_函数功能简述”，确保函数标识统一。

- 函数功能简述需简洁准确，避免冗余，直接说明函数核心用途（如“axk\_uart\_init：初始化串口外设，配置波特率、数据位等参数”）。

- 参数说明需明确，对于指针参数，需注明是否为输入/输出参数、是否允许为空；对于枚举、结构体参数，需简要说明核心字段含义。

- 返回值说明需全面，尤其是有多个返回值的函数，需逐一说明每个返回值对应的场景（如“返回0：初始化成功；返回\-1：参数错误；返回\-2：外设未就绪”）。

- 开发者姓名需填写实际编码人员姓名，日期填写函数首次开发日期，若后续修改函数，需在@note中注明修改人、修改日期及修改内容，无需修改函数头其他固定字段。

- 静态函数（static修饰）的函数头，除上述要求外，需在@note中注明“静态函数，仅本文件内可调用”。

- 中断服务函数的函数头，需在@brief中注明中断触发方式（如“外部中断触发”“定时器溢出中断”），@note中注明中断优先级、中断处理注意事项（如“禁止在中断中调用耗时函数”）。

### 1\.3 错误示例及正确示范

错误示例（缺失axk前缀、信息不全）：

```c
// 初始化串口
void uart_init(int baud)
{
    // 函数体
}
```

正确示范：

```c
/*
 * axk_uart_init：初始化串口外设
 * @brief  配置串口的波特率、数据位、停止位、校验位，使能串口接收/发送功能
 * @param  baud：串口波特率，取值范围：9600、19200、38400、115200
 * @return 无
 * @note   1. 调用前需确保串口外设已使能；2. 波特率需与外设匹配，否则通信失败
 * @author 张三
 * @date   2026-04-27
 */
void axk_uart_init(int baud)
{
    // 函数体
}
```

## 二、通用编码规范

### 2\.1 命名规范

- 函数名：必须以“axk\_”为前缀，采用小写字母\+下划线命名法（snake\_case），见名知意，避免缩写（必要缩写需在函数头@note中说明），如axk\_gpio\_set\_level（GPIO电平设置）、axk\_adc\_read\_value（ADC值读取）。

- 变量名：全局变量以“g\_axk\_”为前缀，静态变量以“s\_axk\_”为前缀，局部变量无需前缀，均采用小写字母\+下划线命名，避免单字母变量（除循环变量i、j、k外），如g\_axk\_system\_status（全局系统状态）、s\_axk\_temp\_buf（静态临时缓冲区）。

- 宏定义：全大写字母\+下划线命名，以“AXK\_”为前缀，如AXK\_UART\_BAUD\_115200、AXK\_GPIO\_PIN\_0。

- 枚举、结构体：枚举名、结构体名以“Axk”为前缀，采用驼峰命名法（CamelCase），枚举值、结构体成员采用全大写\+下划线（前缀与枚举/结构体名一致），如typedef enum \{AXK\_STATUS\_OK, AXK\_STATUS\_ERROR\} AxkStatus;。

### 2\.2 代码格式规范

- 缩进：采用4个空格缩进，禁止使用Tab键缩进，确保不同编辑器打开时格式一致；代码块（if、for、while、函数体等）的左大括号“\{”紧跟语句末尾，右大括号“\}”单独占一行，与对应语句对齐。

- 空行：函数之间空2行，函数内部不同逻辑块之间空1行，宏定义、头文件包含与代码主体之间空1行，避免代码过于拥挤。

- 空格：运算符（\+、\-、\*、/、=、==、\!=等）前后各加1个空格；逗号“,”后加1个空格；括号“\(\)”前后不加空格（函数名与括号之间无空格），如if \(a == b\)、axk\_func\(param1, param2\)。

- 换行：每个语句单独占一行，避免多个语句写在同一行；if、for、while等语句的条件判断结束后，即使只有一行函数体，也必须加\{\}包裹，避免逻辑漏洞。

### 2\.3 注释规范

- 除axk函数头注释外，代码中关键逻辑、复杂算法、异常处理、临时修改处，需添加行注释（// 注释内容），注释需简洁准确，与代码同步更新，避免注释与代码不一致。

- 禁止添加无意义注释（如// 这里是循环、// 赋值操作），禁止注释掉代码（无用代码直接删除，需保留的临时代码需在注释中说明保留原因及删除时间）。

- 头文件注释：每个\.h文件开头，需添加文件说明注释，说明文件用途、包含的函数/宏定义、作者、创建日期，格式参考函数头，无需axk前缀。

### 2\.4 安全性规范

- 参数校验：所有函数的输入参数，必须进行合法性校验（如参数是否为空、取值是否在合理范围），避免因非法参数导致程序崩溃、外设异常。

- 内存管理：嵌入式环境中，避免使用动态内存分配（malloc、free），若必须使用，需严格控制内存大小，确保分配成功后及时释放，防止内存泄漏；禁止使用野指针、空指针。

- 中断安全：中断服务函数中，禁止调用耗时函数、禁止进行大量数据处理，尽量缩短中断执行时间；中断与主程序共享变量时，需使用volatile修饰，避免编译器优化导致数据异常。

- 宏定义安全：宏定义中若包含多个表达式，需用括号包裹，避免优先级问题导致逻辑错误，如\#define AXK\_ADD\(a,b\) \(\(a\)\+\(b\)\)。

### 2\.5 可维护性规范

- 函数拆分：单个函数功能需单一，避免“大函数”（函数体代码超过100行，特殊场景除外），复杂功能拆分为多个子函数，子函数仍需遵循axk函数头规范。

- 代码复用：相同功能的代码，封装为工具函数，避免重复编码；常用宏定义、枚举、结构体，统一放在公共头文件中，便于复用和维护。

- 兼容性：代码编写需考虑不同硬件平台的兼容性，避免直接使用硬件寄存器地址（优先使用封装好的宏定义），便于后续产品移植。

## 三、文件组织规范

- 文件命名：\.c文件、\.h文件均以“axk\_”为前缀，采用小写字母\+下划线命名，与文件中核心函数功能一致，如axk\_uart\.c、axk\_uart\.h。

- 头文件包含：\.h文件需添加防重复包含宏（\#ifndef AXK\_XXX\_H \#define AXK\_XXX\_H \.\.\. \#endif），宏名与文件名一致（全大写）；\.c文件仅包含必要的头文件，避免冗余包含。

- 文件结构：\.c文件按“头文件包含 → 宏定义 → 全局变量定义 → 静态函数声明 → 函数定义”的顺序组织；\.h文件按“防重复包含宏 → 宏定义 → 枚举/结构体定义 → 函数声明”的顺序组织。

## 四、执行与考核要求

1. 所有新开发代码，必须严格遵循本规范，尤其是axk函数头规范，代码提交前需自行检查，确保符合要求。

2. 历史代码优化时，需同步修改函数头，补充axk前缀及相关信息，逐步实现全代码规范统一。

3. 代码评审时，编码规范符合性（重点是axk函数头）作为必查项，不符合规范的代码，需退回修改，直至符合要求后方可通过评审。

4. 若发现未按规范编码的情况，将纳入个人工作考核，多次违规将进行内部通报，确保规范落地执行。

## 五、补充说明

1\. 本规范自发布之日起执行，后续若有调整，将提前通知各位同事。

2\. 若在编码过程中遇到特殊场景（如无法遵循某条规范），需提前向我报备，说明原因，经同意后可特殊处理。

3\. 鼓励各位同事提出规范优化建议，共同完善部门编码体系，提升研发效率和代码质量。
